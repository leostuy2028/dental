MMOral-Bench CLOSED-ENDED (multiple-choice) inference prompt — provenance record
================================================================================
CORRECTED 2026-07-05. An earlier version of this file WRONGLY claimed the closed-ended
prompt is VLMEvalKit's base `ImageMCQDataset.build_prompt` ("Please select the correct
answer from the options above."). It is NOT. The MMOral benchmark ships its own eval
script that overrides the base class with a longer custom instruction. That mistake nearly
led us to believe our (correct) reproduction prompt was fabricated — recording it here so
it does not recur.

TRUE SOURCE — used by BOTH evaluation paths, so it is the prompt for the whole leaderboard
(verified verbatim on 2026-07-05, github.com/isbrycee/OralGPT):
  (A) DOCUMENTED path (README: `python run.py --config config_mmoral_opg.json`): the
      `MMOral_OPG_CLOSED` dataset class in MMOral-Bench-EvalKit/vlmeval/dataset/mmoral.py
      (line 122, build_prompt). This is the class the config uses for EVERY model, open or
      API. Config settings: temperature=0, max_tokens=8192, image_detail=high, via the
      dmxapi.cn proxy — these MATCH our reproduction (clients/gpt_client.py) exactly.
  (B) A standalone GPT-4o-only proxy script, MMOral-Bench-EvalKit/eval_MMOral-OPG-Closed.py
      (run_inference_gpt4o), at temperature=0.2.
Both (A) and (B) build the IDENTICAL long prompt below, so whichever path scored GPT-4o's
published 45.40%, the prompt is the same one we reproduce. (A) even matches our generation
settings including temperature=0; only the secondary script (B) uses 0.2.

`MMOral_OPG_CLOSED.build_prompt` (path A) is:

    options_prompt = 'Options:\n'
    for i, letter in zip(range(1, 5), ['A','B','C','D']):
        options_prompt += f"{letter}. {str(row[f'option{i}'])}\n"
    prompt = (f'Question: {row["question"]}\n{options_prompt}'
              + 'Please answer the above multiple-choice question by selecting the single '
                'correct option (A, B, C, or D). If the provided information is insufficient '
                'to determine a clear answer, please choose the most likely correct option '
                'based on the available data and your judgment.')

VERBATIM prompt sent to the model (no system persona; image is attached separately):

    Question: {question}
    Options:
    A. {option1}
    B. {option2}
    C. {option3}
    D. {option4}
    Please answer the above multiple-choice question by selecting the single correct option
    (A, B, C, or D). If the provided information is insufficient to determine a clear answer,
    please choose the most likely correct option based on the available data and your judgment.

Two things about this prompt matter for the audit (PAPER_DRAFT §5.2):
  1. It specifies the TASK ("select the single correct option") but NOT the OUTPUT FORMAT.
     It never says "output only the letter" / "answer with the letter directly" (unlike
     VLMEvalKit's own MMMU-Pro / MUIR / VMCBench prompts, which do). It even invites reasoning
     ("based on the available data and your judgment"). So capable models answer in PROSE.
  2. Answer extraction is `get_single_choice_prediction` in vlmeval/dataset/utils/mmoral_opg.py
     (verified verbatim). It parses the option letter from free-form output, keeps the
     candidate appearing EARLIEST in the text, and FALLS BACK TO A RANDOM LETTER when it cannot
     parse one. On the prose answers the prompt elicits, it misreads ~1 item in 9 (§5.2).

Generation config: the DOCUMENTED path (config_mmoral_opg.json) uses temperature=0,
max_tokens=8192, image_detail=high — identical to our reproduction, so there is NO temperature
deviation from the path the paper documents. (The secondary standalone script uses temp=0.2;
a minor internal inconsistency in the benchmark, not in our reproduction.)

!! FLAGGED DISCREPANCY (blank options) — needs a dedicated look for §5.1 !!
`MMOral_OPG_CLOSED.post_build` runs `self.data[col].fillna('None')` on the option columns
BEFORE build_prompt. So in the real benchmark a blank option is shown to the model as the
literal word "None" (e.g. "D. None"), which reads like a plausible "none of the above" choice.
Our harness instead renders a blank option via str(NaN) as "nan" (e.g. "D. nan"). This is a
REAL difference in what the model sees on the 38 blank-option items, and could change their
behavior (a model may well pick "D. None" as "none of the above"). §5.1's blank-answer
analysis and the reproduction on those items should be re-checked under "None" rendering.
(The standalone script path may differ again — verify before relying.)

Our harness's "coax" prompt (dental-radiologist persona + "output only the letter") DIVERGES
from this on purpose, to make every answer a bare letter the benchmark's own parser reads
exactly. `prompts/gpt.py` implements both as selectable modes.
